Mini Coding Agent

Structure — 一次任务的完整流程

按时间顺序追踪一个 user prompt 从进 runner 到 end_task 收尾的完整路径。 每个阶段标清楚:在哪个文件、谁调用谁、写了哪些数据、读了哪些数据。

5 阶段全景

从上到下按时间顺序;阶段 4 是带循环的主体。

阶段 1 · Runner 入口
runners/mbpp_task.py 读 prompt.md,新建 Environment + MemoryManager,调 engine.run_task
prompt 进 engine
阶段 2 · engine 启动 task
memory.begin_task() 创建 WorkingMemory · memory.get_context_for_planner() 拿 facts + project_context
engine 调 planner
阶段 3 · planner 出 plan
planner LLMNode 跑一次 LLM(max_steps=1, 无 tools) → parse 成 plan_steps(list[str]) → working.set_plan
engine 逐 step 派给 coder
阶段 4 · 主循环(retry + replan)
外层 replan ≤ MAX_REPLANS · 中层遍历 plan_steps · 内层 retry ≤ MAX_RETRIES_PER_STEP
每次 retry 三动作:coder 用 tools 改 workspace → verifier 调 test_runner → engine 看 verdict 决定
所有 step 过完 / 彻底失败
阶段 5 · end_task 收尾
working_memory.json 落盘 · task_history 追加 · passed=True 时 candidate_facts 升级到 long-term + global facts
关键认知:engine 是流程总管,决定 retry / replan / 结束。coder 不直接看 verifier 输出 —— verdict 回到 engine,engine 把 fix_suggestion 拼进下一轮 prompt 给 coder。一句话:
"engine 调度,planner / coder 想,tools / test_runner 翻译,env 干活,memory 旁观记录。"

逐阶段展开

1. Runner 入口:读 prompt.md

通过 python -m runners.mbpp_task 启动,逐个 case 加载 prompt 喂给 engine。

命令
python -m runners.mbpp_task all --exp baseline --limit 1
文件
runners/mbpp_task.py
函数
cmd_run(args) → run_one(workspace, facts_file)
读 prompt
prompt = open(workspace/prompt.md).read() — 不是 terminal 输入,是从数据集物化出来的题面
下一跳
engine.run_task(prompt, planner, coder, memory, metrics)
说明
verifier 不是 LLMNode,是一个纯 pytest 函数(verifier.verify(memory, env)),所以不出现在参数列表里。env 由 coder.env 复用,不再读全局 config.WORKSPACE
↓

2. 任务初始化

生成 task_id,创建 WorkingMemory,拿历史记忆上下文。

文件
engine.py
函数
run_task()
关键调用
memory.generate_task_id() memory.begin_task(task_id, user_prompt) memory.get_context_for_planner()
生成变量
task_id working memory_context failure_context = None overall_passed = False
memory_context
从 long-term memory 读出 project_context + 按 confidence 排序的 top-N facts + 最近几条 task_history,拼成字符串给 planner
↓

3. Planner 生成 plan

把用户任务 + memory_context + failure_context 合并成 prompt,交给 planner LLM,解析为 step 列表。

调用链
planner_role.create_plan(node, ...) → LLMNode.run → llm.chat
输入
user_task memory_context failure_context
特点
planner LLMNode 不带 tools,max_steps = 1 —— 一次 LLM 调用直接出 plan,然后 parse_plan(text) 拆成 list
输出
plan_steps: list[str]
写回
working.set_plan(plan_steps)
注意
planner / coder 是同一个 LLMNode 类的不同实例(role / system_prompt / tools 是构造参数);verifier 不是 LLMNode,是纯函数。"agent" 指的是 engine + 多个 LLMNode + verifier + memory 整体,不是单个 LLMNode
↓

4. 逐 step 调度执行

对 planner 给出的每个 step 逐一执行,每个 step 都会经历 coder → verifier → retry / continue。

外层循环
for replan in range(MAX_REPLANS+1) 最多 3 次 plan
中层循环
for step_idx, step_desc in enumerate(plan_steps)
内层循环
for attempt in range(MAX_RETRIES_PER_STEP) 每个 step 最多 retry 2 次
恢复机制
单 step retry → 整 plan replan → 都失败 → 返回 failure
↓

5. Coder 执行当前 step

coder 是带工具的 LLMNode 实例,跑 ReAct loop。每次启动前先把 working memory snapshot 拼到 prompt 前面,让它知道前面 step 学到了什么。

调用链
coder_role.run_coder(coder, step, memory) → LLMNode.run
注入 snapshot
memory.get_working().snapshot_for_coder() 拼到 user 消息前面 — 包含 plan / 之前 step 的 observations / files_changed
LLM loop
LLMNode.run() 内部循环最多 MAX_STEPS=8 步,每步 llm.chat() → 解析 tool_calls → execute_tool(name, args, env, memory)
tool 调度
tool 函数纯净,只依赖 Environment 和(对 save_memory 而言)memory,不写 working memory
事件落地
每次 LLM 往返 / tool_call / tool_result / final text 都由 LLMNode 循环 push 到 working.event_log
派生视图
working.observations() — 由 event_log 中的 read_file/list_dir tool_result 派生 working.files_changed — 由 event_log 中的 write_file/replace_in_file 派生
输出
{text, completed}
↓

6. Verifier 跑 pytest(纯函数,不调 LLM)

verifier 不是 LLMNode,是一个纯函数。从 memory 拿 test_command,subprocess 跑 pytest,看 returncode 直接出结论。失败时把 stderr 末尾喂给 coder 重试。

调用
verifier.verify(memory, env=coder.env) — env 显式传入,不再读全局 config.WORKSPACE
真跑测试
test_runner.run_tests(env, ...) → env.run_command(command, timeout=...)
test_command
3 级 fallback: memory.project_context.test_command → marker(pytest.ini/pyproject.toml/tests/) → skip
判定
result.passed() = executed AND not timed_out AND returncode == 0 — 没有 LLM 介入
挂掉时
fix_suggestion = (stderr 或 stdout)[-1500:] — pytest 失败堆栈末尾,喂给 coder 下一轮重试
输出
{passed, reason, fix_suggestion, test_block}
↓

7. 处理 verify 结果

PASSED → 进入下一个 step。FAILED → retry 当前 step;retry 用尽 → replan;replan 也用尽 → 整 task 失败。

PASSED
step_passed = True → break inner retry loop
FAILED (反馈层 1)
把 reason + fix_suggestion (pytest stderr 末尾) 拼到 current_step,coder 下一轮直接看到失败堆栈
retry 用尽 (反馈层 2)
_build_failure_context(plan, failed_step_idx, error_history) 生成给 planner 看的完整失败史 → break step loop → 进入下一轮 replan
replan 用尽
overall_passed = False → 返回 "Task failed"
总结
两层反馈循环: step retry(coder 看 verifier 反馈) + plan replan(planner 看完整失败史)
↓

8. end_task 收尾(三文件落盘)

不管成功失败都进入 finally,确保 working memory 落盘 + long-term + global 三个文件都同步。

触发位置
finally 块:try/finally 保证就算异常也一定执行
调用
memory.end_task(task_id, passed=overall_passed, plan, attempts, summary)
passed=True
遍历 wm.candidate_facts → _promote_fact() → 写入 long-term(reinforce 已有 / append 新)
passed=False
candidate_facts 全部丢弃 → 不污染长期记忆
必做
追加 task_history (不含 event_log) → trim → _evict_facts_if_needed() → _save_working_memory() → _save() → drop working memory
三文件写盘
working_memory.json ← 整个 WM 快照(event_log + plan + candidate_facts + files_changed)
long_term_memory.json ← project_context + task_history + 本 case facts (按 source_task_id 过滤)
mbpp_global_facts.json ← 全部 facts(跨 case 累积)

Components — 各层职责与关系

把 tools/ · environment.py · memory.py · test_runner.py 四个核心组件的位置 / 调用关系 / 数据落盘点画清楚。读完之后,任意一次 LLM 决策都能追溯到具体文件 + 写入了哪个磁盘位置。

整体分层

从上往下,箭头是"调用方向"。流程层(engine / planner / coder / verifier)互相调用;基础设施层(tools / memory / test_runner / environment)被流程层使用。

engine.py
Orchestrator — run_task() 编排 plan → execute → verify → retry / replan 主循环。是唯一进入流程层的入口,被 runners/mbpp_task.py 调用。
calls each role
planner.py
LLMNode (max_steps=1, 无 tools) — 拆任务为 step 列表,只调一次 LLM
coder.py
LLMNode (max_steps=8, 带 tools) — ReAct 循环,逐步执行 plan 中的某一步
verifier.py
纯函数,不调 LLM — 跑 pytest 看 returncode,返回 verdict dict
use infrastructure
tools/ & llm.py
tools/ = LLM 看到的"菜单"(schema)+ 执行结果格式化。
llm.py = OpenRouter HTTP 客户端(requests,无 SDK)
memory.py
事件流 + WorkingMemory + 长期 + 全局 facts。
横切层,所有角色 + agent loop 都读写它
test_runner.py
test 领域逻辑:三级 test_command 探测 + TestRunResult 包装。
真正的 subprocess 调用委托给 env.run_command
delegate physical ops to
environment.py
workspace 路径作用域 + read/write/list/walk/run_command 原语。tools 和 test_runner 最终都通过它落地到系统调用 (open / subprocess / os.walk)
横切关系:event_log 怎么传播
LLMNode 循环 (在 llm_node.py 的 LLMNode.run() 里) 每次 LLM 往返 / tool_call / tool_result 都 会主动 push 一条事件到 working memory.event_log。tool 函数本身不 import memory, 所以保持纯净。working.observations() 和 working.files_changed 都是事件流的派生视图,不是单独存的字段。

Tools 层

tools/ — LLM 与底层之间的翻译层

告诉 LLM "你能调哪些函数",并把执行结果格式化成字符串。Tools 不直接做事,委托给 environment / memory。

谁调用
llm_node.py LLMNode.run() 当 LLM 返回 tool_calls 时,通过 execute_tool(name, args, env, memory) dispatch
它调用谁
environment.py 的 read/write/list/walk/run_command (大部分 fs/shell tools)
memory.py WorkingMemory.add_candidate_fact (只有 save_memory tool)
输入
tool 名字 + args dict (llm.py 已把 OpenRouter 返回的 JSON 字符串解析过了)
输出
单个字符串(LLM 友好的格式),如 "[FILE]solution.py\n[FILE]prompt.md" 或 "File not found"
写盘
不直接写盘 — 副作用全部委托给下层。事件由 agent loop push 到 working memory event_log
关键文件
tools/__init__.py TOOL_DEFINITIONS schema + execute_tool dispatcher
tools/fs.py read/write/list/search/replace
tools/shell.py run_command
tools/memory_tool.py save_memory(唯一吃 memory 的工具)
为什么单独抽一层
tools/ 不应该 import subprocess / 知道 docker。哪天换 backend,tools/ 一行不用改

Environment 层

environment.py — workspace 内的物理操作原语

真正去打开文件、跑 subprocess。当前是本地 subprocess + safe_path 路径限制,将来切 docker 只动这一个文件。

谁调用
tools/* 大头 — fs/shell tools 全部委托过来
verifier 间接调用 — 通过 test_runner.run_tests(env, ...) → env.run_command(...)
它调用谁
系统调用 — open(), os.walk(), os.listdir(), subprocess.run()
输入
路径字符串 / shell 命令字符串 / timeout 数字
输出
Python 原生类型 — str · list[dict] · dict。失败抛 Python 异常(交给 tools/ 翻译)
写盘
间接写 — write_file 写 workspace 内文件;不写 memory.json 等元数据文件
路径限制
safe_path() 阻止路径越出 workspace
⚠️ 不是真 sandbox
run_command 是 shell=True + 任意字符串。LLM 想 rm ~ 完全跑得起来。当前 environment 只是路径作用域 + cwd 绑定,不是隔离环境。真要 sandbox 得加 docker / unshare

Memory 子系统

逻辑两层:WorkingMemory(任务级 RAM 对象)+ MemoryManager(持久化层)。 落盘三个文件,每个文件由不同的代码路径写入,关注不同的生命周期。

三个落盘文件

single_case_details/<case>/working_memory.json
每个 case 一个 · end_task 时覆写

装:event_log(每次 LLM 调用 / tool_call / tool_result 的原始流)+ plan + candidate_facts + files_changed + saved_at。

主用途:debug 时回放整个 task 的 LLM 决策。同 task 内多次 replan 共享一个 WorkingMemory,事件累加,最后只 dump 一次。
single_case_details/<case>/long_term_memory.json
每个 case 一个 · end_task 时追加(task_history)

装:project_context + task_history(本 case 的任务记录,不带 event_log) + 本 case 学到的 facts(按 source_task_id 过滤)。

主用途:case 自己的"档案",独立可读,不依赖 working memory 文件。
<exp>/mbpp_global_facts.json
整个实验一个 · 每个 case end_task 时累加

装:跨 case 共享的 facts pool,confidence 累加(同一条 fact 被多个 case 学到 → confidence ↑)。

主用途:跨 case "记忆流动" — planner 启动时按 confidence 拿 top-N 注入 prompt。这是唯一跨 case 流动的存储。

WorkingMemory(任务级 RAM 对象,end_task 时 dump)

memory.py WorkingMemory — 任务进行中的"草稿纸"

begin_task() 创建,end_task() 落盘后丢弃。一个 task 内不管 retry / replan 多少次,共享同一个实例,event_log 持续累加。

谁创建
memory.begin_task(task_id, user_prompt),在 engine.run_task 入口调一次
谁写
agent loop push event_log(llm_call / text / tool_call / tool_result)
save_memory tool add_candidate_fact()
engine set_plan(plan_steps)
谁读
coder 启动每步前读 snapshot_for_coder()(plan + 最近 observations + files_changed)
end_task 读 candidate_facts 决定是否 promote
字段
event_log · plan · candidate_facts · user_prompt · task_id
(派生属性:files_changed / observations 都从 event_log 派生,不存储)
何时落盘
end_task() 调 _save_working_memory(),整个对象 dump 到 working_memory.json
replan 累积
同一 task 内 planner 重新规划时,WorkingMemory 不重建,event_log 继续追加。多次 plan 的所有事件都在同一个 working_memory.json 里

MemoryManager(持久化层,跨 task 存活)

memory.py MemoryManager — 长期记忆的读 / 写中心

每个 case 跑 run_one() 时新建一个,加载 long_term + global,end_task 时把 working memory 落盘 + 推 facts 到 long-term + 同步 global。

构造参数
long_term_file(必传) · working_memory_file(可选) · global_facts_file(可选)
_load() 行为
1. 读 long_term_file → project_context + task_history + facts
2. 若有 global_facts_file → 用 global facts 覆盖 self.data["facts"]
(这是"跨 case 记忆流动"的关键:每个 case 启动时看到的就是上一个 case 留下的 global pool)
_save() 行为
dual-file mode:全部 facts → global file;long_term file 只保留本 case 学的 facts(按 source_task_id 过滤)
single-file mode:全部数据 → long_term file
facts 流转链
coder 调 save_memory tool → WorkingMemory.candidate_facts
→ end_task 时若 passed → _promote_fact() → self.data["facts"]
→ _save() → 同时写 per-case(filtered)+ global(full)
读时谁用
planner 启动前 memory.get_context_for_planner() → 按 confidence 排序的 top-N facts + project_context + 最近 task_history → 拼成 prompt 段落
confidence 累积
新 fact:confidence = 0(FACT_INITIAL_CONFIDENCE)
重复出现:每次 +0.2,封顶 1.0(FACT_REINFORCE_DELTA / FACT_MAX_CONFIDENCE)
evict 策略:超过 MAX_MEMORY_FACTS 时按 (in_grace_period, score, age) 淘汰

Verifier 与 Test Runner

两层分得很干净:verifier 是判决,test_runner 是执行。 verifier 不知道测试是怎么跑的(subprocess / docker),它只读 TestRunResult 字段。 这是 SWE-bench 时升级 docker backend 的关键,verifier 一行不用改。

verifier.py — 判决层(纯函数,不调 LLM)

engine 每跑完一步 coder 都调一次 verify(memory, env)。把 TestRunResult 翻译成给 engine / coder 看的 verdict dict。

谁调用
engine.py run_task() 在每步 coder 跑完后
它调用谁
test_runner.run_tests(env, memory_hint_command, memory_hint_timeout)
输入
memory — 读 project_context.test_command / test_timeout
env — 整个 Environment 实例传给 test_runner;跟 coder 用的是同一个 env,所以测试看到的就是 coder 改完的文件
输出
{passed: bool, reason: str, fix_suggestion: str, test_block: str}
fix_suggestion 哪来
不是 LLM 总结,是 pytest 真实 stderr/stdout 末尾 1500 字。
下一轮 coder LLM 直接读 traceback 自己思考怎么 fix
不在乎什么
不知道 test 是 subprocess / docker / 远程跑的 — 只看 TestRunResult 字段

test_runner.py — test 领域逻辑(命令探测 + 结果包装)

不直接做 subprocess。命令探测 + 结果整理是 test_runner 的事;真正跑命令委托给 env.run_command。

谁调用
verifier.verify()
它调用谁
env.run_command(command, timeout=...) — 全代码库唯一的 subprocess 调用点在 env 内部
test_command 三级 fallback
1. memory.project_context.test_command(memory 提供,优先)
2. marker 探测 — pytest.ini / pyproject.toml / tests/ 任一存在 → "pytest -q"
3. 都没有 → skip,executed=False
输出
TestRunResult(executed, command, returncode, stdout, stderr, timed_out, backend, detection_source, error) — 数据结构,不做判决。backend 字段从 env.backend_name 读
输出截断
stdout/stderr 各超过 4000 字符时,头尾切 + 中间标记 [truncated N chars]
异常处理
env.run_command 抛 subprocess.TimeoutExpired → test_runner 包成 timed_out=True 的 TestRunResult。其他异常包成 executed=False + error 字段
未来扩展
不需要在 test_runner 加 backend 抽象。直接实现 DockerEnvironment(Environment),覆盖 run_command 即可。test_runner / verifier 一行不改
分层回报:重构后 整个代码库只有 environment.run_command 一处 subprocess.run 调用。 tools/shell.py 和 test_runner.py 都通过它走。SWE-bench 加 docker 时,实现 DockerEnvironment 子类、覆盖 run_command, tools / test_runner / verifier 全部 0 行修改。

mbpp_task.py 流程(单文件 runner: setup + run + report)

三个子命令: setup 物化 case → run 跑 agent → all 一条命令搞完。所有 artefact 落在 Execution/<exp_name>/,每次实验自包含。

1. setup: 加载数据集 + 物化 case

从 HuggingFace 拉 MBPP,为每条题写出独立 workspace。

命令
python -m runners.mbpp_task setup --exp baseline --subset sanitized --split test --limit 10
subset
sanitized (默认,427 条人工清洗) 或 full (974 条原始)
数据集字段
sanitized 用 prompt,full 用 text,代码自动判断
缓存
第一次下载到 ~/.cache/huggingface/datasets/,后续秒读
↓

2. materialize_instance: 为每条题写出 4 个文件

workspace + solution.py + test_solution.py + prompt.md + long_term_memory.json

workspace
Execution/<exp>/single_case_details/mbpp_{task_id:04d}/
solution.py
空 stub,等 agent 填代码
test_solution.py
from solution import * + 把 test_list 包成 def test_case_N(): 函数
prompt.md
MBPP 描述 + 展示 test_list 给 agent 看(否则 agent 猜不到函数名)
long_term_memory.json
预填 test_command: "pytest -q test_solution.py" + project_context(language=python, framework=pytest, test_timeout=30)。task_history 和 facts 初始为空
working_memory.json
setup 阶段不预创建。run 阶段每个 task 的 end_task() 写一次,覆盖式
↓

3. 实验目录布局(自包含)

每个实验 = 一个文件夹,删一个文件夹整个实验就没了,可复现。

Execution/
└── <exp_name>/                          # 比如 baseline
    ├── single_case_details/             # 所有 case workspace
    │   ├── mbpp_0011/
    │   │   ├── solution.py              # 空 stub
    │   │   ├── test_solution.py         # pytest harness
    │   │   ├── prompt.md                # 描述 + test 案例
    │   │   ├── long_term_memory.json    # project_context + task_history + 本 case facts
    │   │   └── working_memory.json      # task 跑完后的 WM 快照(event_log + plan + cand. facts)
    │   ├── mbpp_0012/
    │   └── ...
    ├── mbpp_global_facts.json           # 跨 case 共享 facts(confidence 累积)
    └── mbpp_exp_final_results.json      # 结构化 report

run 子命令流程

扫 <exp>/single_case_details/mbpp_*,逐条新建 Environment + MemoryManager,调 engine.run_task,把 report 写到实验目录。

1. 扫描 workspaces

用 glob 找出实验目录下所有 mbpp_xxxx 文件夹。

命令
python -m runners.mbpp_task run --exp baseline [--limit N]
扫描
sorted(glob("Execution/<exp>/single_case_details/mbpp_*"))[:limit]
关键
扫到几个跑几个,setup 阶段物化了什么这里就跑什么
↓

2. 每个 case 新建 Environment(workspace 显式)

不再 mutate 全局变量。env 持有 workspace,后续 verifier 通过 env.workspace 拿到。

代码
env = Environment(workspace, command_timeout=COMMAND_TIMEOUT)
设计要点
workspace 作为构造参数显式传给 env,不 mutate 全局变量。verifier 拿 workspace 走 env.workspace
好处
未来想 --parallel 同时跑多 case 不会撞全局变量
↓

3. 三文件 MemoryManager (per-experiment)

实例化时传 long_term_file + working_memory_file + global_facts_file,启用三文件分离布局。

代码
memory = MemoryManager(
    long_term_file=os.path.join(workspace, "long_term_memory.json"),
    working_memory_file=os.path.join(workspace, "working_memory.json"),
    global_facts_file=facts_file,    # <exp>/mbpp_global_facts.json
)
_load() 行为
先读 long_term_file 拿 project_context + task_history,再读 global file 覆盖 self.data["facts"]
效果
这条 instance 启动时能看到本次实验内所有前面 instance 留下的 facts(不同实验目录互不影响)
三文件分离
working_memory.json 单独放任务级 event_log;long_term_memory.json 不带 event_log(避免数据冗余);global_facts.json 跨 case 共享
↓

4. 调 engine.run_task

用 build_llm_nodes 一次性建好 planner + coder,从 prompt.md 读题面作为 user_prompt,然后调 engine.run_task。

实例化
planner, coder = build_llm_nodes(env, memory, metrics) — verifier 不是 LLMNode,所以不返回
prompt
从 workspace/prompt.md 读出来作为 user_prompt
调用
engine.run_task(prompt, planner, coder, memory, metrics) — 签名没有 verifier 也没有 env(env 走 coder.env 复用)
tool 注入
不需要任何全局 set 函数 — tool dispatcher 直接接收 env / memory 作为参数
↓

5. end_task 双轨 save + 汇总报告

_save() 自动判断模式,facts 写 per-experiment 的 global file,其余写 per-instance。所有 instance 跑完后输出结构化 mbpp_exp_final_results.json。

passed=True
candidate facts → _promote_fact() → 写到 <exp>/mbpp_global_facts.json
passed=False
candidate facts 全部丢弃,task_history 仍然记录
报告状态
passed failed (LLM 答错) crashed (tool/系统错,要修代码)
报告结构
{experiment, model, started_at, finished_at, totals: {total/passed/failed/crashed/other}, results: [...]}
落点
Execution/<exp>/mbpp_exp_final_results.json

三文件 memory 的设计原因

纯 per-instance memory 在 benchmark 模式下让 long-term memory 完全失效

如果每条 MBPP instance 都只有自己的 memory 文件,facts 永远从 0 开始, 跨 case 的 reinforce 不会发生,confidence / reinforce_count / grace period 全部形同虚设。 因此采用"per-case 文件 + 实验级 global facts"的三文件布局。
存储内容目的
Execution/<exp>/single_case_details/<case>/working_memory.json 本 task 完整 event_log + plan + candidate_facts debug 回放;每个 task 独立的"思考过程"快照
Execution/<exp>/single_case_details/<case>/long_term_memory.json project_context + task_history(无 event_log)+ 本 case 学的 facts 看每条 instance 怎么错的、为什么错(便于 debug),case 自己的"档案"
Execution/<exp>/mbpp_global_facts.json 本次实验内所有 instance 共享的 facts(confidence 累积) 让 reinforce 真的能跨 instance 发生(不同 exp 之间隔离)
研究意义
intra-instance vs cross-instance memory 的取舍本身就是一个可研究的问题。 Reflexion / AFlow / Agent-S 处理方式各不相同,可以做消融实验:无 long-term / per-instance only / cross-instance shared 在 MBPP / SWE-bench 上分别跑。未来论文一个章节。

SWE-bench Verified(P2 规划中)

来源

HuggingFaceprinceton-nlp/SWE-bench_Verified
论文SWE-bench: Can Language Models Resolve Real-World GitHub Issues? (Jimenez et al., 2024)
规模500 条(OpenAI 雇人审核过滤的高质量子集)

每条样本字段

{
  "instance_id": "django__django-12345",
  "repo": "django/django",
  "base_commit": "abc123def456...",          // bug 存在的那个 commit
  "patch": "diff --git a/django/...",        // ground truth, agent 不能看
  "test_patch": "diff --git a/tests/...",    // 验证用 test 代码
  "problem_statement": "When using QuerySet.annotate() ...",
  "FAIL_TO_PASS": ["tests.test_queries.test_annotate_with_subquery"],
  "PASS_TO_PASS": ["tests.test_queries.test_basic_filter", "..."]
}

评分逻辑

修改完 → apply test_patch → 跑所有测试。要求 FAIL_TO_PASS 全部从 fail 变 pass + PASS_TO_PASS 全部仍然 pass。任意一项不满足 → 失败。 比 MBPP 严格得多 — agent 不仅要修对 bug,还不能破坏已有功能。

P2 待新增的文件

runners/swebench_task.py (单文件,跟 mbpp_task.py 同结构)

三个子命令: setup (clone repo + checkout + pull docker 镜像 + 写 memory.json) / run (跑 agent + rolling docker rmi) / all。

数据加载
load_dataset("princeton-nlp/SWE-bench_Verified", split="test")
setup 阶段
git clone https://github.com/{repo}.git <exp>/single_case_details/{instance_id}/ git checkout base_commit docker pull swebench/sweb.eval.x86_64.{instance_id}:latest
test_command
把 docker run --rm -v ws:/testbed image bash -c "..." 整段塞到 long_term_memory.json 的 test_command 字段
run 阶段
扫 <exp>/single_case_details/*,逐条新建 Environment(workspace) + MemoryManager,调 engine.run_task(),verifier 拿到 docker test_command 后 env.run_command 直接跑(整个 docker 调用就是普通 subprocess)
rolling cleanup
每条跑完后 docker rmi 删掉这条 instance 的镜像,节省磁盘
实验目录
跟 MBPP 一样: Execution/<exp>/{single_case_details, swebench_global_facts.json, swebench_exp_final_results.json}
verifier 不需要任何改动
因为 P0 verifier 设计成"只接收一个 test_command 字符串然后 subprocess 跑", SWE-bench 的 docker 命令就是把 docker run ... 整段塞到 test_command 里, verifier 完全感知不到。这就是 P0 设计的回报 — P2 阶段 verifier 改 0 行代码。
