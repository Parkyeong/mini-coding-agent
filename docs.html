Mini Coding Agent

Structure — 主流程

Mini Coding Agent 的运行主流程:从 main() 接收用户输入, 到 end_task() 收尾的完整链路。每张卡片对应一个阶段,点击"展开细节"看具体函数和关键代码。

1. 接收用户输入

程序从 CLI 主循环开始,等待 user prompt。

文件
main.py
函数
main()
关键代码
user_input = input("\n User:").strip()
下一跳
run_task(user_input, planner, coder, verifier, memory, metrics)
↓

2. 任务初始化

生成 task_id,创建 WorkingMemory,拿历史记忆上下文。

文件
main.py
函数
run_task()
关键调用
memory.generate_task_id() memory.begin_task(task_id, user_prompt) memory.get_context_for_planner()
生成变量
task_id working memory_context failure_context = None overall_passed = False
↓

3. Planner 生成 plan

把用户任务 + memory_context + failure_context 合并成 prompt,交给 planner LLM,解析为 step 列表。

调用链
planner.create_plan → BaseAgent.run → llm.chat
输入
user_task memory_context failure_context
特点
Planner 不带 tools,max_steps = 1 —— 一次 LLM 调用直接出 plan
输出
plan_steps: list[str]
写回
working.set_plan(plan_steps)
↓

4. 逐 step 调度执行

对 planner 给出的每个 step 逐一执行,每个 step 都会经历 coder → verifier → retry / continue。

外层循环
for replan in range(MAX_REPLANS+1) 最多 3 次 plan
中层循环
for step_idx, step_desc in enumerate(plan_steps)
内层循环
for attempt in range(MAX_RETRIES_PER_STEP) 每个 step 最多 retry 2 次
恢复机制
单 step retry → 整 plan replan → 都失败 → 返回 failure
↓

5. Coder 执行当前 step

Coder 继承 BaseAgent.run(),区别是它带工具,可以进入 tool-calling loop。每次 run 之前先注入 working memory snapshot。

调用链
coder.reset_message() → coder.run(current_step)
注入 snapshot
memory.get_working().snapshot_for_coder() 拼到 user 消息前面
LLM loop
BaseAgent.run() 内部循环最多 MAX_STEPS=8 步,每步 llm.chat() + tool 调用
tool 副作用
read_file → _record_observation write_file → _record_file_change save_memory → wm.add_candidate_fact
输出
{text, completed}
↓

6. Verifier 真跑测试 + 判定

verifier 先从 memory 拿 test_command 真跑一遍,把 returncode/stdout/stderr 作为 [TestRunner] block 喂给 LLM 做综合判定。

调用
verifier.verify(user_prompt, step_desc, coder_result, files_changed)
真跑测试
_maybe_run_test() → test_runner.run_tests() → SubprocessRunner.run()
test_command
3 级 fallback: memory → marker → skip
LLM 判定
verifier 把 [TestRunner] block + coder 自报 + files_changed 一起喂给 LLM,要求 STATUS: PASSED|FAILED
输出
{passed, reason, fix_suggestion}
↓

7. 处理 verify 结果

PASSED → 进入下一个 step。FAILED → retry 当前 step;retry 用尽 → replan;replan 也用尽 → 整 task 失败。

PASSED
step_passed = True → break inner retry loop
FAILED
把 reason / fix_suggestion 拼到 current_step,再 retry 一次
retry 用尽
_build_failure_context() 生成给 planner 看的失败摘要 → break inner step loop → 触发 replan
replan 用尽
overall_passed = False → 返回 "Task failed"
↓

8. end_task 收尾

不管成功失败都进入 finally,确保 working memory 一定被 promote 或丢弃,task_history 一定写盘。

触发位置
finally 块:try/finally 保证就算异常也一定执行
调用
memory.end_task(task_id, passed=overall_passed, plan, attempts, summary)
passed=True
遍历 wm.candidate_facts → _promote_fact() → 写入 long-term(reinforce 已有 / append 新)
passed=False
candidate_facts 全部丢弃 → 不污染长期记忆
必做
追加 task_history → trim → _evict_facts_if_needed() → _save() → drop working memory
双轨模式
_save() 自动判断:有 global_facts_file → 把 facts 写到 global,其余写 per-instance

Mini Coding Agent 计划在两个 benchmark 上评测:MBPP(P1,简单)和 SWE-bench Verified(P2,真实工程难度)。

数据集对比

维度MBPPSWE-bench Verified
本质写一个独立函数修复真实开源项目里的 bug
难度入门级真实 senior dev 难度
样本数sanitized 427(test split 257)500(人工筛选)
每条 size几 KB几十 ~ 几百 MB(整个 repo)

测试运行环境本地 pytestdocker 容器
任务描述长度1 句话几百字 + stack trace

5. end_task 双轨 save + 汇总报告

passed=False

报告状态
passed failed (LLM 答错) crashed (tool/系统错,要修代码)

双轨 memory 的设计原因

per-instance memory 在 benchmark 模式下让 long-term memory 完全失效

每条 MBPP instance 都有自己的 memory.json,跑 mbpp_0602 时 task_history 从 [] 开始, 跑 mbpp_0603 时又是新的 memory.json,facts 永远从 0 开始,从来不会 reinforce。 设计文档里的 confidence / reinforce_count / grace period 全部形同虚设。
存储内容目的
Execution/mbpp_xxxx/memory.json project_context + task_history 看每条 instance 怎么错的、为什么错(便于 debug)
Execution/mbpp_global_facts.json 所有 instance 共享的 facts 让 reinforce 真的能跨 instance 发生
研究意义
intra-instance vs cross-instance memory 的取舍本身就是一个可研究的问题。 Reflexion / AFlow / Agent-S 处理方式各不相同,可以做消融实验:无 long-term / per-instance only / cross-instance shared 在 MBPP / SWE-bench 上分别跑。未来论文一个章节。

SWE-bench Verified(P2 规划中)

来源

HuggingFaceprinceton-nlp/SWE-bench_Verified
论文SWE-bench: Can Language Models Resolve Real-World GitHub Issues? (Jimenez et al., 2024)
规模500 条(OpenAI 雇人审核过滤的高质量子集)

每条样本字段

{
  "instance_id": "django__django-12345",
  "repo": "django/django",
  "base_commit": "abc123def456...",          // bug 存在的那个 commit
  "patch": "diff --git a/django/...",        // ground truth, agent 不能看
  "test_patch": "diff --git a/tests/...",    // 验证用 test 代码
  "problem_statement": "When using QuerySet.annotate() ...",
  "FAIL_TO_PASS": ["tests.test_queries.test_annotate_with_subquery"],
  "PASS_TO_PASS": ["tests.test_queries.test_basic_filter", "..."]
}

评分逻辑

修改完 → apply test_patch → 跑所有测试。要求 FAIL_TO_PASS 全部从 fail 变 pass + PASS_TO_PASS 全部仍然 pass。任意一项不满足 → 失败。 比 MBPP 严格得多 — agent 不仅要修对 bug,还不能破坏已有功能。

runners/run_swebench.py

跟 run_mbpp.py 类似的 for 循环 + rolling 模式镜像清理。

扫描
glob("Execution/<repo>__<repo>-*")
复用 main 流程
同样调 run_task(),verifier 拿到 docker test_command 后 SubprocessRunner 直接跑

每条跑完后 docker rmi 删掉这条 instance 的镜像,节省磁盘

因为 P0 verifier 设计成"只接收一个 test_command 字符串然后 subprocess 跑", SWE-bench 的 docker 命令就是把 docker run ... 整段塞到 test_command 里, verifier 完全感知不到。这就是 P0 设计的回报 — P2 阶段 verifier 改 0 行代码。
